# Вопросы для комиссии

## Как реализованы популярные товары на главной?

### Простыми словами
Сейчас это не аналитика по заказам и не случайные товары. На главной просто берутся первые 8 товаров из общего каталога и выводятся как витрина.

### Ответ для комиссии
На главной странице блок популярных товаров реализован как витринный вывод первых позиций каталога. На текущем этапе он не рассчитывается по истории заказов и не формируется случайным образом.

## Где генерируется QR-код и как он попадает на сайт и в WPF?

### Простыми словами
Сам код получения создаётся в API при оформлении заказа. Для сайта сам QR-код-картинка тоже генерируется в API. В WPF картинка QR строится уже в клиенте на основе того же кода получения, который пришёл от API.

### Ответ для комиссии
Уникальный код получения формируется на сервере в API при создании заказа. В веб-версии изображение QR-кода возвращается отдельным API-методом. В WPF-приложении изображение QR формируется локально из уже полученного кода, поэтому и сайт, и WPF используют один и тот же серверный код получения.

## Как формируется код получения и почему он уникальный?

### Простыми словами
Код формируется методом на сервере в формате вроде `MP-123456`. После генерации сервер проверяет, нет ли такого кода уже в базе. Если есть, генерирует заново.

### Ответ для комиссии
Код получения создаётся серверным методом в API в момент оформления заказа. Он генерируется в заданном формате и дополнительно проверяется на уникальность через базу данных. Также в самой базе есть уникальный индекс, который не позволяет сохранить дубликат.

## Можно ли заказать товар, если его нет в наличии?

### Простыми словами
Нет, нельзя. Сервер проверяет остаток товара перед добавлением в корзину и ещё раз перед оформлением заказа.

### Ответ для комиссии
Нет, оформление недоступно для товара с нулевым остатком. Проверка выполняется на серверной стороне как при добавлении в корзину, так и при подтверждении заказа, что исключает обход ограничения только за счёт интерфейса.

## Есть ли транзакции, если два человека одновременно закажут последний товар?

### Простыми словами
Полноценной защиты на уровне транзакции сейчас нет. Остаток проверяется и уменьшается, но при очень редком одновременном запросе теоретически возможна гонка.

### Ответ для комиссии
В текущей реализации выполняется серверная проверка остатка перед созданием заказа, однако полноценный механизм конкурентного резервирования через транзакцию или блокировку строки не реализован. Это известная точка для дальнейшего усиления, если систему развивать как промышленную.

## Дополнительные вопросы, которые могут задать

### Как разграничен доступ между ролями?

#### Простыми словами
У каждой роли свои действия: покупатель оформляет и смотрит свои заказы, сотрудник ПВЗ работает только со своим пунктом выдачи, админ управляет системой.

#### Ответ для комиссии
В системе используется ролевая модель доступа. Права пользователя определяются ролью, а на сервере дополнительно проверяется, имеет ли конкретный пользователь право просматривать или изменять нужный заказ.

### Как устроен процесс выдачи заказа?

#### Простыми словами
Сначала заказ создаётся, потом сотрудник ПВЗ отмечает его как доставленный, и только после этого его можно выдать клиенту по коду.

#### Ответ для комиссии
Заказ проходит через последовательность статусов: создание, доставка в ПВЗ и выдача. Выдача разрешена только после подтверждения доставки в пункт выдачи, что исключает преждевременное получение.

### Где хранится фронтенд сайта при релизной сборке?

#### Простыми словами
Сайт собирается отдельно, потом его готовые файлы копируются в `wwwroot` API, и дальше API раздаёт этот фронтенд как обычный сайт.

#### Ответ для комиссии
В релизной конфигурации клиентская часть собирается в статические файлы и встраивается в `wwwroot` серверного проекта. За счёт этого веб-интерфейс можно раздавать непосредственно через API без отдельного фронтенд-сервера.

### Как защищены чужие заказы?

#### Простыми словами
Сервер теперь проверяет, кто именно запрашивает заказ. Просто зная чужой `id` или код выдачи, получить чужой заказ нельзя.

#### Ответ для комиссии
Доступ к заказам контролируется на серверной стороне. API проверяет роль пользователя и принадлежность заказа, поэтому просмотр чужого заказа по прямому идентификатору или коду выдачи запрещён.

## 1. Краткая суть проекта

Разработан дипломный проект информационной системы маркетплейса. Система предназначена для продажи товаров через интернет с возможностью:

- просмотра каталога товаров покупателем;
- регистрации и авторизации пользователей;
- добавления товаров в корзину;
- оформления заказа;
- выбора пункта выдачи заказов;
- отслеживания заказов покупателем;
- получения QR-кода или кода выдачи заказа;
- обработки заказов работником пункта выдачи;
- администрирования товаров, категорий, пунктов выдачи и работников.

Проект разделен на три основные части:

1. **MarketplaceAPI** - серверная часть на ASP.NET Core Web API.
2. **MarketplaceWeb** - веб-клиент для покупателей на React + TypeScript + Vite.
3. **MarketplaceWPF** - desktop-клиент на WPF для администратора и работников пункта выдачи.

Такое разделение демонстрирует клиент-серверную архитектуру и использование одного API несколькими клиентами: веб-приложением и настольным приложением.

## 2. Назначение системы

Система решает задачу автоматизации работы интернет-магазина/маркетплейса с пунктами выдачи заказов. В рамках диплома проект можно описывать как информационную систему, которая объединяет:

- пользовательскую витрину товаров;
- личный кабинет покупателя;
- модуль корзины и оформления заказов;
- модуль управления товарами;
- модуль управления справочниками;
- модуль управления работниками ПВЗ;
- модуль обработки заказов в пунктах выдачи.

Система подходит для демонстрации полного цикла заказа:

1. Покупатель заходит на сайт.
2. Просматривает каталог.
3. Добавляет товар в корзину.
4. Регистрируется или авторизуется.
5. Оформляет заказ.
6. Выбирает пункт выдачи.
7. Получает код/QR-код заказа.
8. Работник пункта выдачи видит заказ в WPF-клиенте.
9. Работник отмечает заказ как доставленный или выданный.
10. Администратор управляет товарами, категориями, пунктами выдачи и работниками.

## 3. Архитектура проекта

Архитектура проекта клиент-серверная.

### 3.1. Серверная часть

Серверная часть реализована в проекте `MarketplaceAPI`. Это ASP.NET Core Web API на .NET 8. API отвечает за:

- подключение к базе данных SQL Server;
- работу с сущностями через Entity Framework Core;
- регистрацию и авторизацию пользователей;
- выдачу JWT-токенов;
- проверку ролей через атрибуты `[Authorize]`;
- обработку HTTP-запросов от Web и WPF клиентов;
- CRUD-операции для товаров, категорий, пунктов выдачи и работников;
- работу с корзиной;
- создание и обработку заказов;
- генерацию QR-кода заказа.

### 3.2. Веб-клиент

Веб-клиент реализован в проекте `MarketplaceWeb`. Это SPA-приложение на React, TypeScript и Vite. Оно предназначено в первую очередь для роли `Customer`, то есть для покупателей.

Веб-клиент отвечает за:

- главную страницу;
- каталог товаров;
- страницу товара;
- регистрацию;
- вход;
- гостевую корзину;
- серверную корзину авторизованного пользователя;
- оформление заказа;
- список заказов;
- страницу деталей заказа;
- отображение QR-кода заказа;
- личный профиль покупателя.

### 3.3. WPF-клиент

WPF-клиент реализован в проекте `MarketplaceWPF`. Это настольное приложение на .NET 8 WPF. Оно используется для внутренних ролей:

- `Admin`;
- `PickupPointWorker`.

Основные задачи WPF-клиента:

- вход пользователя;
- регистрация покупателя;
- управление товарами;
- управление категориями;
- управление пунктами выдачи;
- управление работниками ПВЗ;
- просмотр всех заказов администратором;
- просмотр заказов конкретного ПВЗ работником;
- поиск заказа по QR-коду или коду заказа;
- изменение статуса заказа.

В проекте исторически присутствуют и WPF-страницы покупателя (`CatalogPage`, `CartPage`, `OrdersPage`), но основной покупательский сценарий сейчас ориентирован на веб-клиент `MarketplaceWeb`.

## 6. База данных

Используется Microsoft SQL Server. Доступ к БД идет через Entity Framework Core.

### 6.1. Конфигурация подключения

Подключение настроено правильно через конфигурационные файлы, без хардкода внутри `ApplicationDbContext`.

Локально, на ноутбуке разработчика, используется:

```text
MarketplaceAPI/appsettings.Development.json
Server=DESKTOP-RCL5NLB;Database=MarketplaceDB;Trusted_Connection=True;TrustServerCertificate=True;
```

Для запуска на другом ПК или внешней демонстрации используется Somee:

```text
MarketplaceAPI/appsettings.json
Server=MarketplaceDB.mssql.somee.com;Database=MarketplaceDB;User Id=...;Password=...;TrustServerCertificate=True;Encrypt=True;
```

Важно: при передаче проекта в новый чат или в диплом не нужно раскрывать реальный пароль от Somee. В тексте диплома достаточно написать, что подключение вынесено в конфигурационный файл.

В `launchSettings.json` для профилей `http` и `https` выставлено:

```text
ASPNETCORE_ENVIRONMENT=Development
```

Поэтому при запуске через Visual Studio или командой:

```powershell
dotnet run --launch-profile https
```

локально используется `appsettings.Development.json`, то есть локальная MS SQL база.

## 8. Авторизация и безопасность

Авторизация реализована через JWT.

### 8.1. Регистрация

Endpoint:

```text
POST /api/auth/register
```

Данные:

```json
{
  "email": "user@mail.ru",
  "password": "123456",
  "fullName": "Иванов Иван",
  "phone": "+375..."
}
```

Логика:

- проверяются обязательные поля;
- email приводится к нижнему регистру;
- проверяется корректность email;
- пароль должен быть минимум 6 символов;
- проверяется уникальность email;
- пароль хешируется через BCrypt;
- создается пользователь с ролью `Customer`.

### 8.2. Вход

Endpoint:

```text
POST /api/auth/login
```

Данные:

```json
{
  "email": "user@mail.ru",
  "password": "123456"
}
```

Ответ:

```json
{
  "token": "jwt-token",
  "userId": "guid",
  "email": "user@mail.ru",
  "fullName": "Иванов Иван",
  "role": "Customer",
  "pickupPointId": null
}
```

JWT содержит claims:

```text
NameIdentifier - Id пользователя
Sub - Id пользователя
Email - email
Name - ФИО
Role - роль
```

### 8.3. Разграничение доступа

Для защиты endpoints используются атрибуты:

```csharp
[Authorize]
[Authorize(Roles = "Admin")]
[Authorize(Roles = "PickupPointWorker,Admin")]
```

В веб-клиенте есть `ProtectedRoute`, который пропускает только авторизованного пользователя с ролью `Customer`.

Если пользователь с ролью `Admin` или `PickupPointWorker` пытается войти на сайт, сайт сообщает, что для этих ролей нужно использовать WPF-клиент.

## 12. Сценарии работы системы

### 12.1. Сценарий регистрации покупателя

1. Пользователь открывает сайт.
2. Переходит на страницу регистрации.
3. Вводит ФИО, email, телефон и пароль.
4. Клиент проверяет совпадение паролей.
5. API проверяет обязательные поля, email, уникальность email и длину пароля.
6. Пароль хешируется через BCrypt.
7. В таблице `Users` создается пользователь с ролью `Customer`.
8. Пользователь переходит на страницу входа.

### 12.2. Сценарий входа

1. Пользователь вводит email и пароль.
2. API ищет пользователя по email.
3. Пароль проверяется через BCrypt.
4. Если данные корректны, API формирует JWT.
5. Клиент сохраняет токен.
6. Клиент использует токен в заголовке:

```text
Authorization: Bearer <token>
```

### 12.3. Сценарий гостевой корзины

1. Неавторизованный пользователь открывает каталог.
2. Добавляет товары в корзину.
3. Товары сохраняются в `localStorage`.
4. Пользователь переходит к оформлению.
5. Система просит войти.
6. После входа гостевая корзина переносится в серверную через API.
7. Гостевая корзина очищается.

### 12.4. Сценарий оформления заказа

1. Авторизованный покупатель открывает корзину.
2. Проверяет товары и итоговую сумму.
3. Переходит на checkout.
4. Выбирает ПВЗ.
5. Нажимает "Подтвердить заказ".
6. API проверяет наличие товаров.
7. API создает заказ.
8. API создает позиции заказа.
9. API уменьшает остатки товаров.
10. API очищает корзину.
11. Клиент показывает экран успешного оформления.
12. Покупатель может открыть QR-код или список заказов.

### 12.5. Сценарий обработки заказа работником ПВЗ

1. Работник входит в WPF-клиент.
2. Система определяет его роль `PickupPointWorker` и `PickupPointId`.
3. Работник открывает заказы своего ПВЗ.
4. Система показывает только заказы, привязанные к его ПВЗ.
5. Работник может отметить заказ как доставленный.
6. После прихода покупателя работник ищет заказ по коду/QR.
7. Работник нажимает кнопку выдачи.
8. Статус меняется на `PickedUp`.
9. В заказ записывается время выдачи и Id работника.

### 12.6. Сценарий администрирования

1. Администратор входит в WPF-клиент.
2. Открывает управление товарами.
3. Добавляет новый товар или редактирует существующий.
4. Может указать ссылку на изображение и сразу увидеть предпросмотр.
5. Открывает управление категориями и ПВЗ.
6. Может создать работника и привязать его к ПВЗ.
7. Может просматривать все заказы.

## 13. Работа с изображениями товаров

Изображения товаров не хранятся в базе как файлы. В БД хранится только строковое поле:

```text
Products.ImageUrl
```

Это URL изображения.

Рекомендованный способ:

1. Загрузить картинку на Cloudinary или другой стабильный хостинг изображений.
2. Получить прямую ссылку на картинку.
3. Вставить ссылку в `ImageUrl` через WPF-диалог товара или SQL.

В Web:

- для отображения используется `SafeImage`;
- изображение вписывается через `object-contain`;
- если ссылка битая, показывается заглушка;
- карточки не ломают верстку.

В WPF:

- в `ProductEditDialog` есть поле URL;
- рядом есть предпросмотр;
- при неправильной ссылке показывается подсказка.

## 16. Что важно подчеркнуть в дипломной работе

В дипломе стоит сделать акцент на следующих моментах:

1. Проект реализован как полноценная клиент-серверная информационная система.
2. Один backend обслуживает несколько клиентов: Web и WPF.
3. Есть ролевой доступ:
   - покупатель;
   - администратор;
   - работник ПВЗ.
4. Используется JWT-аутентификация.
5. Пароли не хранятся в открытом виде, а хешируются BCrypt.
6. База данных нормализована:
   - пользователи отдельно;
   - товары отдельно;
   - категории отдельно;
   - заказы и позиции заказов отдельно;
   - ПВЗ отдельно.
7. История заказа сохраняет цену на момент покупки через `PriceAtOrder`.
8. Реализована работа со статусами заказов.
9. Реализован QR-код/код получения заказа.
10. Реализована гостевая корзина и перенос в серверную корзину после входа.
11. Интерфейсы разделены по пользователям:
    - покупатель работает через сайт;

## 24. Короткое резюме проекта одним абзацем

Дипломный проект представляет собой клиент-серверную информационную систему маркетплейса, реализованную на ASP.NET Core Web API, React и WPF. Серверная часть обеспечивает работу с SQL Server, авторизацию пользователей через JWT, управление товарами, категориями, пунктами выдачи, корзиной и заказами. Веб-клиент предназначен для покупателей и позволяет просматривать каталог, регистрироваться, входить в систему, работать с корзиной, оформлять заказы и получать QR-код заказа. WPF-клиент предназначен для администратора и работников пунктов выдачи: администратор управляет товарами, категориями, ПВЗ и работниками, а работник ПВЗ обрабатывает заказы и выдает их покупателям по коду/QR-коду. Проект демонстрирует ролевую модель доступа, работу с базой данных, REST API, хранение паролей в виде хешей, формирование заказов и разделение интерфейсов под разные категории пользователей.
